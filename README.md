# Jokes-journal
---
### **Функциональные требования**

**Проект: "Журнал приколов"**
**Цель:** Веб-платформа для публикации и обсуждения юмористических контентов (приколов).
**Технологический стек:** Backend - Python, Django; Frontend - HTML (скорее всего, с использованием Django Templates), CSS.

---

#### **1. Функционал регистрации, аутентификации и авторизации**
*   **1.1. Регистрация нового пользователя.**
    *   Система должна предоставлять форму для регистрации с полями: `имя пользователя`, `email` (опционально), `пароль`, `подтверждение пароля`.
    *   Пароль должен храниться в базе данных в захэшированном виде.
    *   После успешной регистрации пользователь автоматически авторизовывается в системе.
*   **1.2. Вход (аутентификация) пользователя.**
    *   Система должна предоставлять форму для входа с полями: `имя пользователя` (или `email`), `пароль`.
    *   При вводе неверных учетных данных система должна сообщать об ошибке.
*   **1.3. Выход пользователя из системы.**
    *   В интерфейсе залогиненного пользователя должна быть кнопка/ссылка "Выйти", завершающая сеанс работы.
*   **1.4. Ограничение доступа.**
    *   Неавторизованные пользователи могут только просматривать список постов и детальные страницы постов с комментариями.
    *   Создавать, редактировать и удалять посты и комментарии могут только авторизованные пользователи.

#### **2. Функционал управления постами**
*   **2.1: Просмотр списка всех постов.**
    *   Система должна отображать на главной странице список всех опубликованных постов, отсортированных по дате создания (новые сверху).
    *   Для каждого поста в списке должна отображаться краткая информация: `заголовок`, `автор`, `дата публикации`, `количество комментариев`, `превью содержания`, `категория`.
*   **2.2: Просмотр детальной страницы поста.**
    *   При переходе на пост система должна отображать полное содержимое поста, всю информацию об авторе и дате, а также все комментарии.
*   **2.3: Создание нового поста.**
    *   Авторизованный пользователь должен иметь доступ к форме создания поста.
    *   Форма должна содержать поля: `заголовок` (обязательное), `содержание` (обязательное), `категория` (обязательное), `локация` (опционально).
    *   После успешного сохранения пользователь перенаправляется на детальную страницу созданного поста.
*   **2.4: Редактирование существующего поста.**
    *   Авторизованный пользователь может редактировать **только те посты, автором которых он является**.
    *   На странице просмотра своего поста должна быть кнопка "Редактировать".
    *   Редактирование происходит через форму, предзаполненную текущими данными поста.
*   **2.5: Удаление поста.**
    *   Авторизованный пользователь может удалять **только те посты, автором которых он является**.
    *   При удалении поста также должны удаляться все связанные с ним комментарии.

#### **3. Функционал управления комментариями**
*   **3.1: Добавление комментария к посту.**
    *   На детальной странице поста должна быть форма для добавления комментария.
    *   Форма доступна только авторизованным пользователям и содержит одно поле: `текст комментария` (обязательное).
*   **3.2: Просмотр комментариев.**
    *   Все комментарии к посту должны отображаться на его детальной странице, обычно отсортированные по дате (сначала старые или сначала новые - на ваше усмотрение, но лучше указать).
    *   Для каждого комментария отображается: `автор`, `дата публикации` и `текст`.
*   **3.3: Редактирование комментария.**
    *   Авторизованный пользователь может редактировать **только свои комментарии**.
    *   Ряд с комментарием должна иметь кнопку "Редактировать" (для его автора).
*   **3.4: Удаление комментария.**
    *   Авторизованный пользователь может удалять **только свои комментарии**.

#### **4. Общий функционал и интерфейс**
*
    *   На всех страницах сайта должна быть единая панель навигации с ссылками: "Главная", "Правила".
    *   На панели навигации должна отображаться информация о текущем пользователе: для неавторизованных - кнопки "Войти"/"Регистрация", для авторизованных - кнопки "Написать пост" и "Выйти", а также кнопка с именем пользователя.
    *   На странице пользователя должны быть аватар, никнейм, список постов.
    *   На детальной странице поста должна быть возможность перейти по названию категории и увидеть все посты, относящиеся к этой категории.

### **Сценарии использования**

**Акторы:**
*   **Гость (Неавторизованный пользователь)**
*   **Авторизованный пользователь**
*   **Автор (частный случай Авторизованного пользователя, который создал контент)**

---

#### **Сценарий 1: Регистрация нового пользователя**

1.  **Актор:** Гость.
2.  **Цель:** Создать новую учетную запись для получения доступа к полному функционалу сайта.
3.  **Предусловия:** Пользователь не авторизован в системе.
4.  **Основной поток событий:**
    1.  Пользователь переходит на главную страницу.
    2.  Пользователь нажимает кнопку "Регистрация" в навигационной панели.
    3.  Система отображает форму регистрации.
    4.  Пользователь заполняет поля `имя пользователя`, `email` и `пароль`, подтверждает пароль.
    5.  Пользователь нажимает кнопку "Зарегистрироваться".
    6.  Система проверяет данные на валидность (уникальность логина, сложность пароля, совпадение паролей).
    7.  Система создает новую учетную запись, хэширует пароль и сохраняет данные в БД.
    8.  Система автоматически авторизовывает пользователя.
    9.  Система перенаправляет пользователя на главную страницу с сообщением об успешной регистрации.
5.  **Альтернативный поток A: Неверные данные**
    1.  На шаге 6 система обнаруживает, что данные невалидны (например, логин уже занят).
    2.  Система отображает форму снова с соответствующими сообщениями об ошибках.
    3.  Сценарий продолжается с шага 4.

---

#### **Сценарий 2: Публикация нового поста**

1.  **Актор:** Авторизованный пользователь.
2.  **Цель:** Опубликовать новый юмористический пост на сайте.
3.  **Предусловия:** Пользователь авторизован в системе.
4.  **Основной поток событий:**
    1.  Пользователь нажимает кнопку "Добавить прикол" в навигационной панели.
    2.  Система отображает форму создания поста.
    3.  Пользователь заполняет поля `заголовок` и `содержание`.
    4.  Пользователь нажимает кнопку "Опубликовать".
    5.  Система проверяет данные на валидность (поля не пустые).
    6.  Система сохраняет новый пост в базе данных, автоматически проставляя автором текущего пользователя и текущую дату и время.
    7.  Система перенаправляет пользователя на детальную страницу созданного поста.
5.  **Альтернативный поток A: Данные невалидны**
    1.  На шаге 5 система обнаруживает ошибку (например, поле `заголовок` пустое).
    2.  Система отображает форму снова с сообщением об ошибке, подсвечивая проблемное поле.
    3.  Сценарий продолжается с шага 3.

---

#### **Сценарий 3: Добавление комментария к посту**

1.  **Актор:** Авторизованный пользователь.
2.  **Цель:** Оставить комментарий к понравившемуся посту.
3.  **Предусловия:** Пользователь авторизован и находится на детальной странице поста.
4.  **Основной поток событий:**
    1.  Пользователь прокручивает страницу к форме добавления комментария.
    2.  Пользователь вводит текст своего комментария в текстовое поле.
    3.  Пользователь нажимает кнопку "Добавить комментарий".
    4.  Система проверяет, что поле не пустое.
    5.  Система сохраняет комментарий в базе данных, привязывая его к текущему посту и устанавливая автором текущего пользователя.
    6.  Система обновляет страницу, и новый комментарий появляется в списке.
5.  **Альтернативный поток A: Пустой комментарий**
    1.  На шаге 4 система обнаруживает, что поле пустое.
    2.  Система отображает сообщение об ошибке под полем ввода.
    3.  Сценарий продолжается с шага 2.

### Бизнес-план проекта «Журнал приколов»
### Концепция
«Журнал приколов» — это площадка для лёгкого юмора, мемов и коротких историй. Цель — создать русскоязычное коммьюнити, похожее на Reddit / Pikabu, но с акцентом на юмор и пользовательское творчество.
### Целевая аудитория
* Возраст: 16–40 лет.
* География: Россия, СНГ.
* Интересы: юмор, развлечения, интернет-культура, мемы.
* Поведение: регулярно потребляют лёгкий контент, активны в соцсетях.
### Уникальное торговое предложение
* Минималистичный и быстрый сайт без лишних функций.
* Сообщество с доброжелательным и юмористическим контентом.
* Возможность быстро запостить шутку — даже с телефона.
### Конкурентный анализ
| Конкурент                 | Особенности             | Слабые стороны                                 |
| ------------------------- | ----------------------- | ---------------------------------------------- |
| Pikabu                    | Крупное сообщество      | Перегруженность контентом, «тяжёлая» модерация |
| Reddit (r/funny, r/memes) | Международная аудитория | Англоязычная платформа                         |
Вывод: «Журнал приколов» может занять нишу «обновлённого Pikabu для юмора» — лёгкий формат, минимум политики и рекламы.
### Монетизация
1. Рекламная модель:
   * Баннерная и нативная реклама.
2. Партнёрские интеграции:
   * Сотрудничество с пабликами и юмористическими брендами.
### Этапы реализации
| Этап                          | Срок      | Цель                                  |
| ----------------------------- | --------- | ------------------------------------- |

| Запуск рекламы и монетизации  | 3–5 мес   | Первые доходы                         |
